# Java Backend AGENTS

## Scope

- 本文件用于约束后端 Java 项目的共享规则
- 当前目录不是实际工程, 不写目录地图, 构建命令或模块导览
- 这里只保留每次会话都需要, 且不能仅靠代码推断的规则

## Repo Truth

- 代码仓库是唯一记录系统, 影响开发、评审、排障、交付的知识必须落到 repo 内
- 能用 linter、结构测试、静态检查、CI 强制的规则, 不要只写成自然语言
- 持续性任务应有 versioned 的计划或进度记录
- 讨论中的结论如果影响代码或协作, 必须写入 repo
- 卡住时先补上下文、工具、脚本、约束或文档, 不靠重复硬试推进

## Java Backend Hard Rules

- 新代码默认构造器注入, 禁止新增字段注入
- Controller 只做协议适配, Service/Application 负责业务编排, Repository/Mapper/DAO 只负责数据访问
- 请求对象、响应对象、持久化对象分离, 不直接暴露数据库实体
- 不把业务决策塞进 Controller、Mapper/Repository 或下游 DTO
- 边界层优先使用 `@Validated`、`@Valid` 和 Bean Validation
- 不吞异常, 不静默返回默认成功, 不写无上下文的 `catch (Exception e)`
- `@Transactional` 只放在明确的服务边界, 事务中避免远程调用、MQ 投递和长耗时 IO
- 远程调用显式处理超时、重试、降级、幂等和错误码, 不把下游失败伪装成成功
- SQL 参数必须安全绑定, 禁止拼接用户输入, 查询避免 `SELECT *`
- 日志使用参数化写法, 带足够业务上下文, 不输出敏感信息原文
- 超时、重试、线程池、开关、阈值应可配置, 禁止硬编码密钥、地址、凭证和环境开关
- MQ、定时任务、批处理和异步执行默认考虑重复执行、失败重试、补偿和可观测性
- 新增和修改业务逻辑时, 必须补充对应的中文注释, 标点可用英文, 结尾不需要句号
- 新增复杂业务逻辑默认补测试, 至少覆盖正常路径、参数失败和下游失败

## Review Checklist

- 是否会产生行为回归或兼容性破坏
- 是否覆盖失败路径、边界条件和幂等问题
- 是否把业务逻辑放进了错误层次
- 是否存在事务边界错误、异常吞没或伪成功返回
- 是否缺少日志上下文、配置项或测试
