# Java Backend AGENTS

## Scope

- 本文件用于维护用户级后端 Java 代理规则, 适用于本机任意位置的 Java 后端项目开发、修改、重构、排障与评审
- 本文件不是当前 `.agents` 仓库的项目说明, 不写本仓库目录地图、构建命令或模块导览
- 这里只保留长期有效、跨项目通用, 且不能仅靠代码推断的用户规则
- 具体工程内若存在更近层级的 `AGENTS.md`, 以更近层级规则补充或收紧本文件

## Repo Truth

- 代码仓库是唯一记录系统, 影响开发、评审、排障、交付的知识必须落到 repo 内
- 能用 linter、结构测试、静态检查、CI 强制的规则, 不要只写成自然语言
- 持续性任务应有 versioned 的计划或进度记录
- 讨论中的结论如果影响代码或协作, 必须写入 repo
- 卡住时先补上下文、工具、脚本、约束或文档, 不靠重复硬试推进

## Agent Working Principles

- 编码前先理解上下文: 先读 manifest、配置、包结构、已有实现和测试, 再决定改法
- 不隐藏困惑: 非琐碎任务存在多个合理解释且会影响改动范围时, 先说明假设、权衡或提出澄清
- 简洁优先: 用最少的代码解决当前问题, 不添加未要求的功能、抽象、配置项或未来弹性
- 精准修改: 每一行改动都应能追溯到用户目标, 不顺手重构、格式化或删除无关代码
- 只清理自己造成的影响: 删除因本次改动变得无用的导入、变量、函数和测试夹具, 不主动清理历史死代码
- 匹配项目风格: 遵守当前 Java 版本、Spring Boot 主版本、持久化框架、日志框架、测试框架和命名习惯
- 目标驱动执行: 对非琐碎任务先定义成功标准, 优先用测试、脚本、日志或可复现步骤闭环验证
- 允许轻量判断: 明显的一行修复、拼写修正或纯文档小改不需要完整流程, 但仍要保持范围精确

## Java Backend Hard Rules

- 新代码默认构造器注入, 禁止新增字段注入
- Bean 尽量保持无状态; 必选依赖用 `final` 字段和构造器传入, 可选依赖才允许 setter 或配置注入
- 项目已用 Lombok 时可用 `@RequiredArgsConstructor`; 如果一个类依赖过多, 优先拆分职责
- Controller 只做协议适配, Service/Application 负责业务编排, Repository/Mapper/DAO 只负责数据访问
- Config、Interceptor、Listener、Job、Consumer、Client/Remote/Gateway 应放在清晰包下, 不与核心业务职责混杂
- 请求对象、响应对象、持久化对象分离, 不直接暴露数据库实体
- 不把业务决策塞进 Controller、Mapper/Repository 或下游 DTO
- 字段命名必须表达业务语义, 禁止含义模糊的 `flag`、`type1`、`ext`、`data`
- 对外接口字段一旦发布, 默认按兼容接口处理, 避免随意改名或改变语义
- 边界层优先使用 `@Validated`、`@Valid` 和 Bean Validation
- 业务异常与系统异常分开处理, 对外服务优先提供统一异常出口
- 不吞异常, 不静默返回默认成功, 不写无上下文的 `catch (Exception e)`
- `@Transactional` 只放在明确的服务边界, 只在需要一致性时开启事务
- 事务中避免远程调用、MQ 投递和长耗时 IO; 跨系统一致性优先考虑幂等、补偿、重试和最终一致性
- 远程调用显式处理超时、重试、降级、幂等和错误码, 不把下游失败伪装成成功
- 远程写操作先定义幂等键或补偿方案, 再讨论重试
- SQL 参数必须安全绑定, 禁止拼接用户输入, 查询避免 `SELECT *`
- 排序、分页、过滤条件必须显式; 复杂 SQL 优先保证可读性和可验证性
- 日志使用参数化写法, 带足够业务上下文, 不输出手机号、身份证号、token、密钥、密码等敏感信息原文
- `error` 日志要包含失败原因和关键上下文, 高频循环日志默认降级到 `debug` 或抽样
- 成组配置优先使用 `@ConfigurationProperties`, 不无节制堆 `@Value`
- 超时、重试、线程池、开关、阈值应可配置, 新增配置要说明默认值策略和缺省行为
- 禁止硬编码密钥、地址、凭证和环境开关
- MQ、定时任务、批处理和异步执行默认考虑重复执行、失败重试、补偿和可观测性
- 批处理尽量支持断点续跑、分段执行或失败重试; 异步执行必须保留必要上下文
- 线程池要显式命名、容量清晰、拒绝策略明确; 共享状态必须有明确同步策略
- 慎用 Lombok `@Data`; 对参与比较、去重、缓存键、Map 键的对象显式确认 `equals/hashCode` 语义
- Java 8 项目不要使用高版本语法; Spring Boot 2.x 不混入 `jakarta.*`, Spring Boot 3.x 不回退旧 `javax.*` 习惯
- 新增和修改业务逻辑时, 必须补充对应的中文注释, 标点可用英文, 结尾不需要句号
- 新增复杂业务逻辑默认补测试, 至少覆盖正常路径、参数失败、下游失败和重复执行

## Review Checklist

- 是否会产生行为回归或兼容性破坏
- 是否有未经验证的假设, 是否需要更简单的实现
- 是否存在无关改动、顺手重构或风格漂移
- 是否覆盖失败路径、边界条件和幂等问题
- 是否把业务逻辑放进了错误层次
- 是否存在事务边界错误、异常吞没或伪成功返回
- 是否缺少日志上下文、配置项或测试
- 是否引入了不必要复杂度, 能否用更少代码达成同一目标

## Verification Standard

- 修改代码后优先运行最贴近改动面的测试、编译或静态检查
- 修 bug 时优先补能复现问题的测试或最小验证步骤, 再修到通过
- 改功能时至少验证正常路径、参数校验失败、下游异常或超时、幂等或重复执行
- 无法运行验证时, 明确说明未验证原因、剩余风险和建议命令
