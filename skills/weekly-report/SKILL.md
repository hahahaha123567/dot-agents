# Weekly Report

## 目标

生成一份可直接发送的中文程序员周报，信息来源分两类：

- 用户提供的本周进展、反思、阻塞、未完成事项
- `~/IdeaProjects/` 下相关 Git 仓库在目标周期内的提交记录

Git 记录只作为参考和提醒，不要把 commit message 直接原样堆进"已完成"部分，更不要编造用户没有确认过的业务结果。

## 默认配置

- 仓库根目录：`~/IdeaProjects/`
- Git 用户名：`hahahaha123567`
- 周期定义：周一 `00:00:00` 到周日 `23:59:59`
- 输出语言：中文

如果用户明确给了时间范围、仓库范围或 Git 作者，优先使用用户指定值。

## 周期规则

默认按下面规则推断"本周 / 上周"：

| 当前日期 | 默认统计周期 |
| --- | --- |
| 周一 | 上周一 -> 上周日 |
| 周二到周日 | 本周一 -> 本周日 |

补充规则：

- 如果用户明确说"上周周报"，直接使用上周一到上周日
- 如果用户明确给出日期区间，停止自动推断，按指定区间处理
- 输出中展示人类可读的日期范围，例如：`2026-03-09 ~ 2026-03-15`

## 执行流程

### 1. 先确定范围和约束

先从用户请求中提取：

- 周报周期
- Git 作者
- 是否只看指定项目
- 是否需要面向特定格式或收件人

如果这些信息缺失，按默认值执行；只有当缺失会明显影响结果时再追问。

### 2. 收集用户输入

优先收集下面 4 类信息，缺什么补什么：

1. 本周完成了哪些工作
2. 有哪些反思、优化、沉淀或经验
3. 哪些工作未完成
4. 未完成的原因、阻塞点和下一步计划

如果用户只给了很碎的描述，先帮他整理成要点，再生成完整周报。

### 3. 扫描 Git 仓库

遍历 `~/IdeaProjects/` 下的子目录，只处理 Git 仓库。对每个仓库：

1. 判断是否存在 `.git`
2. 查询目标作者在统计周期内**所有分支**上的提交（不仅限于当前分支）
3. 忽略 merge commit
4. 对跨分支结果去重（同一 commit 可能被多个分支引用，按 commit hash 去重）
5. 记录项目名、提交时间、commit message

建议使用类似命令：

```bash
git log --all --author="hahahaha123567" --since="2026-03-09 00:00:00" --until="2026-03-15 23:59:59" --no-merges --pretty=format:"%H%x09%ad%x09%s" --date=short
```

- `--all` 覆盖所有本地分支和远程追踪分支，而非仅当前 HEAD
- `%H`（commit hash）用于去重，最终展示时去掉 hash
- 如果仓库长期未 fetch 导致远程分支过旧，可先执行 `git fetch --all --quiet` 再查询；非必须，仅在用户反馈记录缺失时补做

处理原则：

- 空仓库、非 Git 目录、无提交目录直接跳过
- 只把 Git 记录当作"线索"，用于补充用户可能遗漏的工作项
- 如果多个 commit 指向同一件事，要合并总结，不要机械逐条罗列
- 同一 commit 出现在多个分支时只记录一次

### 4. 按业务主题聚合

**不要按代码项目（仓库名）聚合工作项。** 一个业务需求往往跨多个仓库，按仓库拆分会让周报读起来像 Git 日志而不是工作汇报。

正确做法：

1. 先从所有仓库的 commit 中识别出业务主题（如"用户权限改造""订单超时优化""新增数据导出功能"）
2. 将同一业务主题下、来自不同仓库的 commit 归到一起
3. 与用户输入合并时，同样按业务主题对齐

聚合示例：

- 错误：项目 A 做了 3 个 commit、项目 B 做了 2 个 commit → 按项目分两组
- 正确：这 5 个 commit 都是"订单超时优化"→ 合并为一条工作项

其他原则不变：

- 用户明确说完成了的事情，优先使用用户表述
- Git 中有但用户没提的内容，可以作为"可补充项"提示用户确认
- 不要把纯技术动作夸大成业务成果
- 反思沉淀要更抽象一些，例如：流程优化、技术选型经验、协作问题、稳定性改进

### 5. 输出最终周报

默认输出以下结构（注意：紧凑排版，段落之间不要多余空行）：

```
## 本周已完成及反思沉淀
- 业务主题 1：完成了什么、达成了什么效果
- 业务主题 2：完成了什么、达成了什么效果
- 反思/沉淀：具体内容
## 本周未完成及原因分析
- 未完成项 1：原因及下一步计划
- 未完成项 2：原因及下一步计划
## Git 提交记录
### 项目 A
- 2026-03-10 commit message
- 2026-03-12 commit message
### 项目 B
- 2026-03-11 commit message
```

如果用户公司有固定模板，按对方模板改写，但保留"已完成 / 未完成 / Git 参考"这三层信息。

## 输出格式要求

最终周报需要直接复制到飞书、企业微信、邮件等外部平台，必须遵守：

- **无多余空行**：标题和列表项之间不加空行，段落之间最多一个空行
- **无行首/行尾多余空格**：每行内容前后不留空格
- **列表项紧凑**：连续的 `-` 列表项之间不插入空行
- **不使用代码块包裹周报正文**：直接输出纯文本/markdown，不要用 ``` 包一层
- **不添加额外的分隔线或装饰符号**

## 生成要求

- 语气自然、像人写的周报，不要像日志导出
- "已完成"按业务主题聚合，写结果和价值，不按仓库拆分，不只写动作
- "未完成"必须带原因，最好有下一步
- 如果本周 Git 很少，不要强行扩写
- 如果本周 Git 很多，先聚类再总结，避免刷屏

## 交互策略

适合少量追问，不要一上来连问很多问题。推荐顺序：

1. 先自动算周期并扫 Git
2. 把初步整理结果给用户看
3. 只追问缺失的关键内容，例如反思、阻塞、未完成项
4. 输出可直接复制的最终版本

## 异常处理

- `~/IdeaProjects/` 不存在：告诉用户仓库目录不存在，请提供正确路径
- 找不到任何 Git 仓库：跳过 Git 参考，仅根据用户输入生成
- 找不到指定作者提交：提醒用户确认 Git 用户名或邮箱
- 用户几乎没提供内容：先给出"基于 Git 的草稿 + 待补充问题"
