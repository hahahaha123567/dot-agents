# Branch Diff Review（面向测试的分支对比 Review）

## 适用场景

- 用户说"比较当前分支和 xxx 分支，输出 code review"
- 用户说"帮我出一份给测试同学的改动说明"
- 用户需要一份面向 QA 的变更摘要，用于设计测试用例

## 约束

- 只读操作，不修改任何代码、不提交、不推送。
- 报告以**测试同学**为主要读者，侧重功能影响和测试建议，弱化纯代码实现细节。
- 输出中文（除非用户明确要求英文）。
- 参考 Codex 官方 review prompt 的缺陷判定方法，但保持本 skill 的中文报告格式：只把可证明、由本次 diff 引入、场景明确、可操作的问题写成缺陷结论；无法证实的问题写成测试关注点。
- **排版**：报告正文减少无意义的空行与重复换行（见「排版与密度」），保持可读即可，勿用连续多个空行堆砌。

## 工作流

### Step 0：确定对比分支

1. 用户消息中通常包含目标分支名（如 `origin/master`、`master-finance`）。
2. 若用户未指定，使用 AskQuestion 工具（如可用）让用户选择；若不可用则询问。
3. 执行 `git fetch` 确保远程分支最新（若失败则跳过，使用本地缓存）。

### Step 1：收集 diff 上下文

**并行执行**以下命令：

```bash
git branch --show-current
git log <target_branch>..HEAD --oneline
git diff <target_branch>...HEAD --stat
git diff <target_branch>...HEAD
```

若 diff 为空，告知用户"当前分支与目标分支无差异"，结束流程。

### Step 2：分析与分类

对 diff 内容按以下维度分析：

1. **功能变更**：新增/修改/删除了哪些业务功能
2. **接口变更**：API、RPC、MQ 消息格式是否有变化
3. **数据变更**：数据库 schema、配置项、枚举值是否有变化
4. **依赖变更**：第三方库、内部 SDK 版本是否有升级
5. **重构/优化**：不影响功能的代码结构调整

### 缺陷与风险判定

- 只有当问题满足以下条件时，才在报告中写成"明确缺陷"：本次 diff 引入；会影响正确性、性能、安全性或可维护性；离散可修；能说明触发条件和受影响路径；作者知道后大概率会修。
- 不把历史遗留问题、个人风格偏好、纯格式问题、无法证明影响范围的猜测写成缺陷。
- 若怀疑影响其他模块，必须通过 diff、调用方、配置或接口契约定位具体受影响对象；否则转化为回归测试建议。
- 若变更像是有意的产品或业务行为调整，除非能证明破坏兼容性、既有契约或关键链路，否则不要按 bug 表述。
- 对明确缺陷可使用 `[P0]`、`[P1]`、`[P2]`、`[P3]` 标注严重性：`P0` 为阻断发布或线上核心使用，`P1` 为下个迭代必须修，`P2` 为普通缺陷，`P3` 为低风险改进。
- 缺陷描述应短而具体，包含文件/行号、触发条件、用户或系统影响；保持本 skill 的中文报告格式。

对每项变更评估风险等级：
- **高风险**：涉及资金/计费/权限/状态机/数据迁移，或改动了核心链路
- **中风险**：改动了已有逻辑的分支条件、增减了判断维度
- **低风险**：新增字段透传、纯重构无逻辑变化、依赖小版本升级

若仅从 diff 无法判断影响范围，使用 Read / Grep 工具查看上下文后再下结论。

### Step 3：输出报告

严格按照下方模板输出；换行与空行遵循「排版与密度」，避免重复换行。

## 输出模板

```markdown
# Code Review 报告（测试视角）
**当前分支**：<branch_name> | **对比基准**：<target_branch> | **提交数**：<N> 个 | **变更文件**：<M> 个（+X 行 / -Y 行）

## 一、改动概述
> 用 2-3 句话说明本次改动的业务背景和目标。

## 二、功能变更明细
> 按功能模块分组，每个模块包含：

### 模块名 / 功能点
- **变更类型**：新增 / 修改 / 删除
- **风险等级**：🔴 高 / 🟡 中 / 🟢 低
- **改动说明**：用非技术语言描述"改了什么、为什么改、对用户/业务有什么影响"
- **影响范围**：列出受影响的上下游系统、页面、接口
- **测试建议**：
  - [ ] 具体的测试用例 1（含输入条件和预期结果）
  - [ ] 具体的测试用例 2

## 三、接口变更摘要
> 若无接口变更则写"本次无接口变更"。

| 接口/方法 | 变更类型 | 说明 |
|-----------|---------|------|
| ...       | 新增参数 / 返回值变化 / 新接口 | ... |

## 四、配置与依赖变更
> 若无则写"本次无配置/依赖变更"。

| 变更项 | 旧值 | 新值 | 影响说明 |
|--------|------|------|---------|
| ...    | ...  | ...  | ...     |

## 五、风险提示与回归建议
> 列出需要重点关注的风险点，以及建议回归测试的范围。
1. **风险点**：描述 + 触发条件 + 建议验证方式；若为明确缺陷，标注 `[P0]`/`[P1]`/`[P2]`/`[P3]` 并给出文件/行号
2. ...
**建议回归范围**：
- [ ] 功能 A 的正常流程
- [ ] 功能 B 的边界场景

## 六、补充说明
> 其他需要测试同学知晓的信息（如特殊环境要求、灰度策略、开关配置等）。若无则写"无"。
```

## 编写指南

### 排版与密度

- **禁止**连续两个及以上空行；章节之间、段落之间**最多空一行**。
- 标题紧接首段内容时，标题与 `>` 引用或正文之间**不额外空行**；**表格前保留一个空行**（便于 Markdown 解析），表格后若无新段落则不要多空行。
- 统计行（当前分支/对比基准等）可合并为**一行**或用顿号/全角空格分隔，避免每条占多行。
- 能用列表表达的不要拆成多个短段落；测试建议用嵌套列表连续列出，条目之间**不要**空行分隔。

### 语言风格

- 面向非开发人员，避免过度使用代码术语。
- 用"当 A 条件满足时，系统会做 B"而非"if(A) then B"。
- 涉及具体字段/枚举时可附代码片段，但需配文字说明。

### 测试用例建议

每个功能变更至少给出 2 条测试建议，覆盖：
- **正向路径**：正常业务流程能走通
- **边界/异常**：空值、极端值、权限不足、并发等
- 若涉及多条件组合（如矩阵式分支），列出关键组合的覆盖表

### 风险等级判定

优先标记以下场景为高风险：
- 计费/扣费逻辑变更
- 权限/鉴权逻辑变更
- 状态机/流程引擎变更
- 数据库 DDL 变更
- 影响多租户的共享逻辑

### 大 diff 处理

变更文件 ≥ 10 个或 diff ≥ 500 行时：
- 按模块分批分析，优先处理业务逻辑层
- 纯格式化/import 排序可归入"补充说明"一笔带过
- 对测试同学影响最大的变更放最前面
